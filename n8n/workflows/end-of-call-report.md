# Workflow 3: End-of-Call Report

**Workflow ID:** `oRACRgI5oTF5TSGk` | **Webhook:** `POST /webhook/vapi-server` | **5 nodes, gated pipeline**

```
Webhook → Parse EOC Envelope → Is EOC? ─┬─ (true)  → Strip Meta → Upsert Interaction
                                         └─ (false) → (nothing — dropped)
```

---

## Node 1: Webhook (trigger)

- Listens for `POST /webhook/vapi-server`
- **Auth:** Header Auth — validates `x-vapi-secret` header against the stored credential (`PQXagW16N8ErYDM7`)
- Uses `lastNode` response mode — the response is sent by whatever node runs last in the chain

**Who calls it:** This is the **assistant-level `serverUrl`** (defined in `assistant_config.json`). VAPI sends *all* lifecycle events to this URL — not just end-of-call reports, but also `assistant-request`, `status-update`, `speech-update`, etc. That's why this workflow needs a gate (the IF node) to filter for the one event type it cares about.

**Key difference from Workflows 1 & 2:** Those workflows listen on *per-tool* server URLs (`/webhook/lookup-caller`, `/webhook/log-interaction`) and only fire when GPT-4o makes a specific tool call. This one receives **every VAPI platform event** for the assistant.

---

## Node 2: Parse EOC Envelope (Code node — JS)

This node does two things:

### a) Check the message type

```js
const messageType = message.type || '';
if (messageType !== 'end-of-call-report') {
  return [{ json: { is_eoc: false } }];
}
```

If the event is *not* an end-of-call report (e.g., it's a `status-update` or `speech-update`), it immediately returns `{ is_eoc: false }` — which the IF node will drop. This is the **early exit** for non-EOC events.

### b) Extract fields from the VAPI end-of-call-report payload

If it *is* an EOC event, it pulls data from two different locations in the VAPI payload:

| Field | Source | Notes |
|-------|--------|-------|
| `caller_name` | `message.analysis.structuredData.callerName` | From the `analysisPlan.structuredDataPlan` — VAPI's own post-call analysis, NOT the LLM's tool call |
| `phone_number` | `message.call.customer.number` | The caller's phone from the VAPI call object |
| `summary` | `message.analysis.summary` | Generated by VAPI's `analysisPlan.summaryPrompt`. **Fallback:** if missing, concatenates all message contents from `message.artifact.messages` (capped at 500 chars) |
| `sentiment` | `message.analysis.structuredData.sentiment` | Validated against `['positive', 'neutral', 'negative']`, defaults to `'neutral'` |
| `timestamp` | Server-generated | `new Date().toISOString()` |
| `vapi_call_id` | `message.call.id` | The VAPI call UUID — the dedup key |

**Output:** `{ is_eoc: true, caller_name, phone_number, summary, sentiment, timestamp, vapi_call_id }`

**Note:** The `callerName` comes from VAPI's `structuredDataPlan` analysis, which is defined in `assistant_config.json`. The structuredDataPlan schema includes `callerName` (added 2026-03-18), so VAPI extracts the caller's full name from the transcript during post-call analysis. If the caller was never identified, the field will be an empty string, and the fallback `|| 'Unknown'` applies.

## Node 3: Is EOC? (IF node)

Simple boolean gate:

```
Condition: $json.is_eoc === true
```

- **True branch** (first output) → Strip Meta
- **False branch** (second output) → empty (no connected nodes — the event is dropped silently)

This is the **filter** that makes this workflow safe to receive all VAPI lifecycle events. Without it, every `status-update` and `speech-update` would try to write to the database.

---

## Node 4: Strip Meta (Code node — JS)

```js
const { is_eoc, ...dbFields } = $input.first().json;
return [{ json: dbFields }];
```

Removes the `is_eoc` flag via destructuring, leaving only the 6 DB-ready fields: `caller_name`, `phone_number`, `summary`, `sentiment`, `timestamp`, `vapi_call_id`.

**Why this exists:** The next node uses `autoMapInputData`, which maps *every* field in the input to a table column. If `is_eoc` were still present, the Supabase node would try to insert it as a column — which doesn't exist in the `interactions` table, causing an error. Same pattern as Log Interaction's toolCallId problem, solved the same way.

---

## Node 5: Upsert Interaction (Supabase node)

- **Operation:** `create` on `interactions` table
- **Data mapping:** `autoMapInputData` — the 6 fields map 1:1 to table columns
- **`alwaysOutputData: true`** — ensures the node always produces output
- **`onError: continueRegularOutput`** — this is the **critical difference from Log Interaction**

### The dedup mechanism

The `interactions` table has a UNIQUE constraint on `vapi_call_id`. Two scenarios:

| Scenario | What happens |
|----------|-------------|
| **Log Interaction didn't fire** (call dropped, LLM timeout, etc.) | This INSERT succeeds — the EOC report is the only record for this call. **Safety net.** |
| **Log Interaction already fired** (happy path) | This INSERT violates the UNIQUE constraint. `onError: continueRegularOutput` silently swallows the error. **No duplicate row.** |

**Why `continueRegularOutput` instead of `continueErrorOutput`?**

Log Interaction uses `continueErrorOutput` because it has two Respond nodes — it needs to tell VAPI whether the write succeeded or failed (GPT-4o is waiting for the tool-call response).

End-of-Call Report uses `continueRegularOutput` because **there's nobody waiting for a response**. This is a fire-and-forget platform event. The response goes back to VAPI's lifecycle event system, which doesn't care about the content. There's no branching — success and failure both just... end. The duplicate rejection is invisible.

## FAQ

### Why does this workflow share the same webhook path as other VAPI lifecycle events?

VAPI only supports **one** `serverUrl` per assistant. Every lifecycle event (`status-update`, `speech-update`, `end-of-call-report`, etc.) goes to the same endpoint. The IF node acts as the router, only processing EOC events and dropping everything else.

### Why not just rely on Log Interaction?

Log Interaction depends on GPT-4o choosing to call the tool. It fails silently when:
- The caller hangs up abruptly (no wrap-up phase)
- The call drops due to network issues
- GPT-4o hits max tokens (500) or the 10-minute `maxDurationSeconds` limit
- GPT-4o simply forgets to call the tool

The end-of-call report fires **for every call, unconditionally**, because it's a platform-level event. It's the safety net.

### What data does the EOC report have that Log Interaction doesn't?

The VAPI `analysisPlan` produces structured metadata that the LLM tool call doesn't capture — specifically `callReason`, `authenticated`, `claimNumber`, `claimStatus`, and `escalated`. However, the current workflow **doesn't write these fields** to the `interactions` table — it only writes the same 6 fields as Log Interaction. The structured data is available in the VAPI payload but isn't persisted yet. This could be a future enhancement.

### What happens in the happy path?

1. Caller says goodbye
2. GPT-4o calls `log_interaction` → Log Interaction workflow writes to `interactions` (succeeds)
3. GPT-4o says goodbye, caller hangs up
4. VAPI sends `end-of-call-report` to `/webhook/vapi-server`
5. Parse EOC Envelope extracts fields, `is_eoc: true`
6. Is EOC? → true → Strip Meta removes `is_eoc`
7. Upsert Interaction tries INSERT → UNIQUE constraint violation on `vapi_call_id`
8. `onError: continueRegularOutput` swallows the error silently
9. Done — no duplicate, no error surfaced

### What happens in the drop/crash path?

1. Caller hangs up abruptly mid-conversation
2. GPT-4o never calls `log_interaction` — no record in `interactions`
3. VAPI still sends `end-of-call-report`
4. Same flow as above, but this time the INSERT **succeeds** — first write for this `vapi_call_id`
5. The call is logged with VAPI's analysis summary instead of the LLM's

## End-to-end example (safety net path)

1. Caller is mid-conversation checking their claim status
2. Caller's phone drops — call ends abruptly
3. GPT-4o never reaches wrap-up, `log_interaction` never fires
4. VAPI detects call ended, runs `analysisPlan`, sends `end-of-call-report` to `/webhook/vapi-server`
5. Parse EOC Envelope: `messageType === 'end-of-call-report'` → extracts `caller_name: "Sarah Johnson"` (or `"Unknown"` if caller was never identified), `phone_number: "+14155551234"`, `summary: "Caller checked status of claim CLM-2024-001..."`, `sentiment: "neutral"`, `vapi_call_id: "abc-123"`
6. Is EOC? → true
7. Strip Meta → removes `is_eoc`
8. Upsert Interaction → INSERT succeeds (no prior record for this call)
9. The call is logged despite the LLM never having a chance to do it
