# System Architecture

> Last verified: 2026-03-18 (Production-ready conversational AI optimization)

## Project Goal

**Vsupport-agent** — VoiceAI insurance claims support agent for "Observe Insurance." VAPI handles voice (Deepgram STT + GPT-4o + ElevenLabs TTS), n8n Cloud handles webhook auth + envelope parsing + all DB operations via its built-in Supabase node (REST API), Supabase PostgreSQL is the data store.

## Current State

**Phases 0–7 complete + Supabase migration + FAQ RAG.** Airtable replaced with Supabase PostgreSQL. n8n workflows query Supabase via the built-in Supabase node (`n8n-nodes-base.supabase`) using the REST API (PostgREST) — no direct Postgres wire connection. Phone normalization for lookups handled in n8n Code nodes (JS-based E.164 normalization). FAQ retrieval uses Supabase pgvector + Edge Functions (gte-small embeddings) with VAPI Custom Knowledge Base integration. 10 passing tests (VAPI envelope parsing). Next: Phase 8 (Technical Write-Up).

## Tech Stack

- **Voice Platform**: VAPI (telephony + STT/TTS/LLM orchestration)
- **STT**: Deepgram Nova-2 (via VAPI)
- **LLM**: GPT-4o (via VAPI)
- **TTS**: ElevenLabs (via VAPI)
- **Workflow Engine**: n8n Cloud — webhook auth, envelope parsing, **all DB operations via built-in Supabase node** (REST API / PostgREST)
- **Data Store**: Supabase PostgreSQL (customers + interactions tables, replaces Airtable)
- **CLI Tools**: VAPI CLI, pyairtable CLI (wrapped via custom `/vapi` and `/airtable` skills)

## Data Flow

```
Caller → VAPI (Deepgram STT → GPT-4o → ElevenLabs TTS)
              ↓ tool calls
         n8n Cloud webhooks (auth + envelope parse)
              ↓ Supabase node (REST API / PostgREST over HTTPS)
         Supabase PostgreSQL
              ├── customers table (read — getAll with PostgREST filters)
              └── interactions table (write + dedup via create with unique constraint)
```

n8n connects to Supabase via the REST API (HTTPS), not the Postgres wire protocol.

### FAQ RAG Data Flow

```
Caller asks FAQ question
  → VAPI detects KB query needed (via knowledgeBaseId)
  → Sends knowledge-base-request to Supabase Edge Function (search-faq)
  → Edge Function: embed query (gte-small) → RPC match_faqs (pgvector) → return docs
  → VAPI LLM uses retrieved docs to answer naturally

FAQ insertion:
  → Insert row into faqs table (no embedding needed)
  → Call generate-embedding Edge Function (or configure DB webhook)
  → Embedding auto-generated and stored (gte-small, 384 dimensions)
```

### Supabase Edge Functions

| Function | Purpose | Trigger |
|----------|---------|---------|
| `generate-embedding` | Auto-embed FAQ rows using gte-small (384 dims) | DB webhook on `faqs` INSERT/UPDATE, or manual invocation |
| `search-faq` | VAPI Custom KB endpoint: embed query → pgvector similarity search → return top 3 docs | VAPI knowledge-base-request |

**Edge Function secrets:** `VAPI_SECRET` (set via `supabase secrets set`). `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` are auto-available.

## Integration Points

- **VAPI → n8n**: Tool-call webhooks with `X-Vapi-Secret` auth header (Header Auth credential)
- **n8n → Supabase**: Built-in Supabase node (`n8n-nodes-base.supabase`) via REST API / PostgREST for all reads and writes. Lookup Caller queries the `phone_digits` generated column (strips `+` from E.164) to avoid the Supabase node's `encodeURI()` mangling of `+`.
- **n8n response wrapping**: Code nodes format results into VAPI's `{ results: [{ toolCallId, result }] }` format

**n8n Workflow Node Patterns:**
- **Lookup Caller** (5 nodes): Webhook → Parse Envelope (Code, builds PostgREST `or=(...)` filter using `phone_digits` column — strips `+` from phone before filtering) → Query Customers (Supabase getAll, `alwaysOutputData: true`) → Format Result (Code, includes `date_of_birth` in single-match response for DOB verification) → Respond
- **Log Interaction** (7 nodes): Webhook → Parse Envelope (Code, outputs DB columns only — no `toolCallId`) → Insert Interaction (Supabase create, `autoMapInputData`, `onError: continueErrorOutput`) → Format Success/Error (Code, gets `toolCallId` from Webhook body directly) → Respond Success/Error
- **End-of-Call Report** (5 nodes): Webhook → Parse EOC Envelope (Code, outputs `is_eoc` flag + DB columns) → Is EOC? (IF, checks `is_eoc === true`) → Strip Meta (Code, removes `is_eoc`) → Upsert Interaction (Supabase create, `autoMapInputData`, `onError: continueRegularOutput` — unique constraint on `vapi_call_id` silently rejects duplicates)

**Phone normalization in n8n:** Lookup Caller's Parse Envelope Code node normalizes US phones to E.164 using JS (strip non-digits, prepend +1 for 10-digit, etc.). Not as robust as the `phonenumbers` Python lib but sufficient for US numbers.

**PostgREST filter for lookups:** The Parse Envelope Code node builds a PostgREST OR filter string (e.g., `or=(phone_digits.eq.14155551234,last_name.ilike.johnson)`). Phone lookups query the `phone_digits` generated column (which strips `+` from `phone_number`) so the filter value never contains `+` — avoiding the Supabase node's `encodeURI()` encoding bug. Uses `ilike` for case-insensitive matching on last_name and claim_number.

## Database Schema (Supabase PostgreSQL)

Schema defined in `supabase/migrations/001_initial_schema.sql` through `005_faqs_table.sql`. Seed data in `supabase/seed.sql` (customers) and `supabase/seed_faqs.sql` (FAQs).

**customers table**: id (auto), first_name, last_name, phone_number (E.164), **phone_digits** (generated, strips `+` from phone_number), claim_number (UNIQUE), claim_status (CHECK: approved/pending/requires_documentation), claim_details, **date_of_birth** (DATE, used for identity verification), created_at.
Indexes: phone_number, **phone_digits**, LOWER(last_name), UPPER(claim_number).

| Name | Phone | Claim # | Status | DOB | Details |
|------|-------|---------|--------|-----|---------|
| Sarah Johnson | +14155551234 | CLM-2024-001 | approved | 1985-03-15 | — |
| Michael Johnson | +14155555678 | CLM-2024-002 | pending | 1990-07-22 | — |
| Emily Chen | +12125559876 | CLM-2024-003 | requires_documentation | 1978-11-04 | Missing proof of loss form and photos of damage |
| David Martinez | +13105554321 | CLM-2024-004 | approved | 1995-01-30 | — |
| Lisa Park | +17185557777 | CLM-2024-005 | pending | 1988-09-12 | — |

Design notes: Two Johnsons for multi-match testing. `requires_documentation` row has `claim_details` populated.

**faqs table**: id (bigserial), question, answer, category (general/claims/billing), embedding (vector(384) — gte-small), created_at. 18 seed entries covering office hours, mailing address, claims process, coverage types, deductibles, filing deadlines, appeals, payment options, cancellation, drivers, proof of insurance, emergencies, departments, processing time, required documents, discounts, contact updates. `match_faqs` RPC function performs pgvector similarity search (inner product `<#>`, threshold 0.8).

**interactions table**: id (auto), caller_name, phone_number, summary, sentiment (CHECK: positive/neutral/negative), timestamp, **vapi_call_id (UNIQUE)**, created_at.
The UNIQUE constraint on vapi_call_id enables `INSERT ON CONFLICT DO NOTHING` for atomic dedup — eliminates the old 10-second wait hack.

## Glossary

| Term | Definition |
|------|-----------|
| **E.164** | International phone number format: `+` followed by country code and number, no spaces/dashes (e.g., `+14155551234`). All phone fields use this. |
| **STT / TTS** | Speech-to-Text / Text-to-Speech. Deepgram Nova-2 handles STT, ElevenLabs handles TTS, both orchestrated by VAPI. |
| **claim_status** | Enum on Customers table: `approved` (claim paid), `pending` (under review), `requires_documentation` (caller must submit missing docs). |
| **Re-verification** | When caller's phone isn't found, the agent asks for last_name → then claim_number to locate the account. Multi-match prompts narrowing. |
| **Containment rate** | % of calls fully resolved by the AI agent without human escalation. Target KPI for voice AI. |
| **AHT** | Average Handle Time — mean duration of a call from pickup to hangup. |
| **CSAT** | Customer Satisfaction score — typically 1–5 post-call rating. |
| **Dedup check** | End-of-call-report uses Supabase create with `onError: continueRegularOutput` — the UNIQUE constraint on `vapi_call_id` silently rejects duplicates. Equivalent to `INSERT ON CONFLICT DO NOTHING`. Previously used a 10s wait + Airtable search. |
| **serverUrl** | VAPI assistant-level webhook that receives lifecycle events (e.g., `end-of-call-report`). Distinct from per-tool server URLs. |
| **Tool call envelope** | The JSON wrapper VAPI sends to n8n webhooks: contains `message.type`, `message.call`, `message.toolCallList[]` with arguments. |
| **Header Auth** | n8n credential type that validates a specific header value (`x-vapi-secret`) on incoming webhooks. |
| **Supabase node** | n8n's built-in node (`n8n-nodes-base.supabase`) for CRUD operations against Supabase via the REST API (PostgREST). Replaced the Postgres node due to IPv6 connectivity issues. |
| **PostgREST** | RESTful API layer built into Supabase that exposes PostgreSQL tables as HTTP endpoints. The Supabase node uses this under the hood. Supports filters like `or=(col.eq.val,col.ilike.val)`. |
| **alwaysOutputData** | n8n node-level setting that ensures at least one empty item is output even when a query returns 0 rows. Prevents downstream nodes from being skipped. |
| **phone_digits** | Generated column on `customers` table: `replace(phone_number, '+', '')`. Stores digits only (e.g., `14155551234`). Lookup Caller filters on this column instead of `phone_number` to avoid the Supabase node's `encodeURI()` mangling `+` in PostgREST filter strings. Indexed via `idx_customers_phone_digits`. |
| **HIPAA** | Health Insurance Portability and Accountability Act. VAPI's `hipaaEnabled: true` activates BAA-eligible mode (no call recording/logging by VAPI). |
| **DOB verification** | Date of birth verification step (Section 2b of system prompt). After name confirmation, the agent asks for the caller's date of birth and compares against the `date_of_birth` field returned by `lookup_caller`. Two attempts allowed before escalation. The stored DOB must never be read back to the caller. |
